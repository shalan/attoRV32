# AttoRV32 Debug Facility — Specification

## 1. Overview

AttoRV32 has no built-in debug module (no DM/DTM, no JTAG). Instead, a
full GDB session — halt, continue, single-step, software breakpoints,
register/memory read/write — is achieved with:

- **Zero core RTL changes** beyond the existing `dbg_halt_req` port.
- A **combinational ROM** (~2.5 KiB) synthesised as LUT logic containing
  the ISR entry and GDB RSP engine.
- A **polled UART** for GDB ↔ target communication.
- A 4-bit **UART break detector** driving `dbg_halt_req`.

Total silicon cost: ~2,300 cells (ROM) + ~200 cells (UART) + 4 FFs
(break detector) + 1 FF (`dbg_halt_mask` in core).

---

## 2. Hardware Architecture

### 2.1 Memory Map

The debug SoC (`rtl/attorv32_dbg.v`) uses `ADDR_WIDTH = RAM_AW + 1`.
The MSB of the address selects ROM (=1) vs RAM/IO (=0).

```
    Example: RAM_AW = 12 → ADDR_WIDTH = 13 (8 KiB total)

    0x0000 ┬───────────────────┐
           │       RAM          │  user .text + .data + .bss + stack
           │                    │  + stub mutable data (g_regs[], BPs, RSP buf)
    0x0FF0 ├───────────────────┤
           │       I/O          │  4 × 32-bit registers (16 bytes)
    0x1000 ├───────────────────┤  ← MTVEC_ADDR
           │    stub ROM        │  _isr_entry + gdb_stub (combinational)
    0x1FFF └───────────────────┘
```

### 2.2 I/O Register Map

| Offset | Name | Access | Description |
|---|---|---|---|
| `+0x0` | `UART_DATA` | R/W | Write: TX byte. Read: RX byte. |
| `+0x4` | `UART_STATUS` | RO | Bit 0: TX ready. Bit 1: RX valid. |
| `+0x8` | *(reserved)* | — | — |
| `+0xC` | `CTRL` | WO | Write any value → `$finish` (bench only). |

### 2.3 UART Break Detector

```verilog
reg [3:0] brk_cnt;
always @(posedge clk)
   if (!resetn)          brk_cnt <= 0;
   else if (uart_rx)     brk_cnt <= 0;         // RX idle → reset
   else if (~(&brk_cnt)) brk_cnt <= brk_cnt+1; // count while low, saturate

wire uart_break = &brk_cnt;  // → dbg_halt_req
```

When `uart_rx` stays low for 15+ clock cycles (a serial break), the
detector asserts `dbg_halt_req`. GDB sends a serial break on Ctrl-C.
The counter self-clears once RX goes high (GDB resumes sending bytes).

### 2.4 Trap Behaviour of `dbg_halt_req`

`dbg_halt_req` bypasses both `mstatus.MIE` and `mcause`:

| Property | Value |
|---|---|
| Blocked by MIE=0? | **No** |
| Blocked by mcause=1 (in handler)? | **No** |
| `mcause` reported | `0x0000_0003` (same as EBREAK) |
| `mepc` saved | `PC_new` (the instruction *about* to execute) |
| Trap vector | `MTVEC_ADDR` (ROM base) |

This means the CPU can be halted at any time, even inside an ISR with
interrupts disabled. The stub sees `mcause=3` and handles it identically
to a software breakpoint.

**Re-trigger prevention:** The core contains a 1-FF `dbg_halt_mask`
register. It is set when `dbg_halt_req` is accepted and cleared on
`mret`. While set, further `dbg_halt_req` assertions are ignored.
This prevents the break detector's level signal from re-triggering
the trap while the ROM stub is executing. Cost: 1 FF + 2 gates.

### 2.5 Stub ROM (`rtl/stub_rom.v`)

A parameterised Verilog module containing a single `always @(*)
case(addr)` block. Auto-generated by `scripts/gen_stub_rom.py` from
compiled RISC-V code. Yosys synthesises it as a combinational LUT tree.

- **Zero latency**: instruction fetches from ROM need no wait states.
- **Read-only**: only `.text` and `.rodata` live in ROM. All mutable
  state (`g_regs[]`, breakpoint table, RSP buffer) lives in RAM.

Measured size: ~2.5 KiB code → 623 words → ~2,300 cells / 13,044 µm² on
Sky130 (area-optimized) with a combinational delay of ~1.3 ns.

## 3. Software Architecture

### 3.1 Components

| File | Section | Description |
|---|---|---|
| `sw/crt0_stub.S` | `.isr` (ROM) | ISR entry: save all regs → `g_regs[]`, call stub, restore, `mret`. |
| `sw/gdb_stub.c` | `.text` (ROM) | RSP packet engine + command handlers. ~600 lines of C. |
| `sw/gdb_stub.h` | — | `g_regs[]` frame definition, entry point, UART API. |
| `sw/gdb_uart.c` | `.text` (ROM) | Platform UART: `gdb_uart_getc()` / `gdb_uart_putc()`. |
| `sw/stub_link.ld.in` | — | Linker script: `.text` in ROM, `.data`/`.bss` in RAM. |

### 3.2 Register Frame (`g_regs[]`)

```c
// RV32I: 33 words = 132 bytes  (x0..x31 + pc)
// RV32E: 17 words =  68 bytes  (x0..x15 + pc)
uint32_t g_regs[NGDB_REGS];
```

Layout matches GDB's g-packet register order:

| Index | Register | Notes |
|---|---|---|
| 0 | `x0` | Always 0 (not saved; slot used as scratch during ISR entry) |
| 1 | `x1` (ra) | |
| 2 | `x2` (sp) | Saved *before* clobbering sp with stub stack |
| … | … | |
| 31 | `x31` (RV32I only) | |
| 32 (or 16) | `pc` | Loaded from `mepc` CSR on entry; written back on exit |

### 3.3 ISR Entry Sequence (`crt0_stub.S`)

The ISR entry is the first code in ROM (at `MTVEC_ADDR = ROM_BASE`):

```
1. sw  sp, __sp_save(zero)       // stash original sp using zero-base addressing
2. la  sp, g_regs                // sp = base of register frame
3. sw  x1, 4(sp)                 // save all registers except x0 and x2
   sw  x3, 12(sp)
   ...
   sw  x31, 124(sp)              // (x16..x31 only for RV32I)
4. lw  t0, __sp_save(zero)       // recover original sp
   sw  t0, 8(sp)                 // save to g_regs[2]
5. csrr t0, mepc                 // save PC
   sw  t0, 128(sp)               // g_regs[32] (or 64(sp) for RV32E g_regs[16])
6. csrr a0, mcause               // argument to gdb_stub_entry()
7. la  sp, __stub_stack_top      // switch to stub's own stack
8. call gdb_stub_entry           // → RSP loop (may modify g_regs[])
9. la  sp, g_regs                // restore base
10. lw  t0, 128(sp)              // restore mepc (GDB may have changed it)
    csrw mepc, t0
11. lw  x1, 4(sp)                // restore all registers
    lw  x3, 12(sp)
    ...
    lw  x31, 124(sp)
12. lw  sp, 8(sp)                // restore original sp (must be last)
13. mret
```

### 3.4 RSP Packet Engine

**Framing:** `$payload#XX` where `XX` is a 2-hex-digit modular checksum.
The stub sends `+` (ACK) or `-` (NAK) after receiving each packet.

**Minimum packet set:**

| Packet | Handler | Reply |
|---|---|---|
| `?` | Stop reason | `T05` (SIGTRAP) |
| `g` | Read all registers | hex-encoded `g_regs[]`, little-endian per word |
| `G` | Write all registers | `OK` |
| `m addr,length` | Read memory | hex-encoded bytes |
| `M addr,length:XX…` | Write memory | `OK` |
| `c [addr]` | Continue | *(exit RSP loop → `mret`)* |
| `s [addr]` | Single-step | *(arm step BPs → exit → re-enter → `T05`)* |
| `Z0,addr,kind` | Insert SW breakpoint | `OK` |
| `z0,addr,kind` | Remove SW breakpoint | `OK` |
| `qSupported` | Feature negotiation | `PacketSize=200` |
| `qAttached` | Attached to process? | `1` |
| `Hg` | Set thread | `OK` (single-threaded) |
| everything else | — | `` (empty packet = unsupported) |

### 3.5 Software Breakpoints

The stub maintains a table of ~10 slots:

```c
struct bp { uint32_t addr; uint32_t orig; uint8_t len; uint8_t used; };
```

**Insert (`Z0`):** Read the instruction at `addr`. If `(instr & 3) == 3`,
it's a 32-bit instruction: save 4 bytes, overwrite with `ebreak`
(`0x00100073`). Otherwise save 2 bytes, overwrite with `c.ebreak`
(`0x9002`). Mark slot as used.

**Remove (`z0`):** Find the slot, restore the saved bytes, free the slot.

**Note:** Software breakpoints only work in RAM. ROM-resident code cannot
be breakpointed (but there is no need to debug the debugger).

### 3.6 Software Single-Step

No hardware stepper is needed. The stub emulates single-step by
**next-PC prediction**:

1. Decode the instruction at `mepc`.
2. Compute every possible successor PC (fall-through + branch/jump target).
3. Plant "step scratch" breakpoints (last 2 slots in the BP table) at each.
4. Return from the stub (→ `mret`).
5. The CPU executes one instruction and immediately hits one of the scratch
   breakpoints → re-enters the stub.
6. Stub detects re-entry from step (a `stepping` flag), removes the scratch
   BPs, sends `T05` stop-reply.

**Instructions requiring branch-target prediction:**

| Instruction | Successor PCs |
|---|---|
| `JAL` | `pc + J-imm` (always taken) |
| `JALR` | `(g_regs[rs1] + I-imm) & ~1` (always taken) |
| `BEQ/BNE/BLT/BGE/BLTU/BGEU` | `pc + 4` AND `pc + B-imm` (both — don't evaluate condition) |
| `c.j` / `c.jal` | `pc + CJ-imm` |
| `c.jr` / `c.jalr` | `g_regs[rs1]` |
| `c.beqz` / `c.bnez` | `pc + 2` AND `pc + CB-imm` |
| All other instructions | `pc + len` (2 or 4 bytes) |

For JALR, the target depends on `rs1` — read from `g_regs[]` which was
saved at trap entry. This is correct: the register values reflect the
state at the moment of the halt.

### 3.7 Async Halt (Ctrl-C)

When the user presses Ctrl-C in GDB, GDB sends a serial break on the
UART line. The break detector in `attorv32_dbg.v` asserts `dbg_halt_req`.
The CPU traps to `MTVEC_ADDR` (ROM). The stub enters the RSP loop and
sends `T05`. GDB regains control.

No IRQ routing, no special UART mode, no protocol changes. The same
TX/RX pair carries both RSP packets and the halt signal.

```bash
# 1. Compile the stub and generate the ROM Verilog:
python3 scripts/gen_stub_rom.py \
    --rom-base 0x1000 --ram-aw 12 --keep

# 2. Synthesise the debug SoC:

# 3. Build user firmware (link for ADDR_WIDTH=13, stub is in ROM):
make -C sw ADDR_WIDTH=13 RV32E=0 HAVE_M=1

# 4. Connect GDB:
riscv64-unknown-elf-gdb firmware.elf
(gdb) target remote /dev/ttyUSB0
(gdb) break main
(gdb) continue
(gdb) info registers
(gdb) step
```

## 6. Limitations

| Limitation | Workaround |
|---|---|
| No data watchpoints | Not supported. Would need trigger CSRs + comparators. |
| `dbg_halt_req` overwrites mepc | Save mepc immediately in ISR entry. `dbg_halt_mask` (1 FF) prevents re-triggering until mret. |
| Software breakpoints only in RAM | ROM code cannot be breakpointed. Place all debuggable code in RAM. |
| No flash breakpoint support | For external flash, use HW breakpoint registers (not implemented). |
| Single-step overhead | ~10 µs per step at 72 MHz (decode + 2 BP plant/remove + 2 UART packets). Acceptable for interactive debugging. |

## 7. Verilator GDB Bridge — Workflow

This section describes the end-to-end workflow for running a live GDB
session against the AttoRV32 debug SoC in a Verilator simulation.

### 7.1 Prerequisites

| Requirement | Tested Version | Notes |
|---|---|---|
| **RISC-V GCC toolchain** | `riscv64-unknown-elf-gcc` | Cross-compiler, assembler, linker, `objcopy`, `objdump` |
| **RISC-V GDB** | `riscv64-elf-gdb` | Your binary name may differ (`riscv64-unknown-elf-gdb`, etc.) |
| **Verilator** | 5.038 | Version 5.x required (`rootp->` signal access, `--public-flat-rw`) |
| **Python 3** | 3.8+ | For `gen_stub_rom.py` (stub compilation) and `gdb_rsp_demo.py` |
| **Make, C++ compiler** | GNU Make, clang/g++ | Verilator compiles a C++ model that must be built |

### 7.2 Build Steps

All commands run from the repository root.

```bash
# Step 1 — Build the Verilator GDB bridge (one command does everything):
bash sim/build_gdb_bridge.sh
```

This script performs three sub-steps automatically:

1. **Generate the stub ROM** (`scripts/gen_stub_rom.py`):
   compiles `sw/crt0_stub.S` + `sw/gdb_stub.c` into a RISC-V ELF,
   extracts the `.text` section, and generates `rtl/stub_rom.v` — a
   combinational Verilog `case` block (~623 words).

2. **Build test firmware** (`sw/dbg_test.c`):
   a minimal counter-loop program that increments `*(0x8)` in a tight
   loop. Compiled to a word-oriented hex file for `$readmemh`.

3. **Verilate + compile**:
   runs `verilator --cc --exe` on the debug SoC (`attorv32_dbg.v` +
   `attorv32.v` + `stub_rom.v` + `tb_dbg_gdb.cpp`), then `make` to
   produce the simulation binary.

Output binary: `build/gdb_bridge/obj_dir/Vattorv32_dbg`

### 7.3 Running a GDB Session

**Terminal 1 — Start the simulation:**

```bash
build/gdb_bridge/obj_dir/Vattorv32_dbg +hex=build/gdb_bridge/fw.hex
```

The simulation prints:

```
[gdb-bridge] Reset released. Running at localhost:3333
[gdb-bridge] Waiting for GDB on localhost:3333 ...
```

It blocks here, waiting for a TCP connection.

**Terminal 2 — Connect GDB:**

```bash
riscv64-elf-gdb build/dbg_test/dbg_test.elf \
    -ex 'target remote localhost:3333'
```

On connect, the bridge triggers a UART break to halt the CPU. GDB
receives a `T05` stop-reply and is ready for interactive debugging.

**Example GDB session:**

```
(gdb) info registers pc sp
pc  0x10   0x10 <_start+16>
sp  0xbf0  0xbf0

(gdb) x/1xw 0x0008          # Read the counter
0x8:  0x00000000

(gdb) set {int}0x0008 = 0x1000   # Write the counter
(gdb) x/1xw 0x0008
0x8:  0x00001000             # Verified

(gdb) break *0x000E          # Software breakpoint
Breakpoint 1 at 0xe

(gdb) continue               # Resume execution
Breakpoint 1, 0x0000000e in _start ()

(gdb) stepi                  # Single-step
0x00000010 in _start ()

(gdb) stepi
0x00000012 in _start ()

(gdb) delete breakpoints
(gdb) continue               # Resume
^C                            # Ctrl-C → async halt
Program received signal SIGINT
(gdb) x/1xw 0x0008
0x8:  0x00015145             # Counter advanced
```

## 8. Constraints and Known Limitations

### 8.1 Verilator Bridge Write-Back Race

**Constraint:** When poking `uart_rx_data` / `uart_rx_valid` into
the DUT's internal registers from the Verilator C++ testbench, new
bytes must NOT be injected while the CPU is in `S_WAIT` (state 3).

**Cause:** AttoRV32 performs register file write-back during both
`S_EXECUTE` and `S_WAIT`. During `S_WAIT` after a load from
`UART_DATA`, `mem_addr` still points to the UART I/O register.
Changing `uart_rx_data` during `S_WAIT` corrupts `LOAD_data` →
`writeBackData` → the register file is overwritten with the new
byte instead of the byte the CPU actually requested.

**Symptom:** Every byte consumed from the UART reads as the *next*
byte instead of the current one. Checksums never match. The stub
sends NAK (`-`) and retries, or loops forever in `get_packet()`.

**Fix in `tb_dbg_gdb.cpp`:**

```cpp
// Only inject when uart_rx_valid == 0 AND state != S_WAIT (3)
if (!uart_rx_valid
    && state != 3  /* S_WAIT */
    && break_cycles == 0) {
    uart_rx_data  = next_byte;
    uart_rx_valid = 1;
}
```

**Scope:** This constraint applies only to the Verilator testbench's
register-poking approach. Real UART hardware with a proper serdes
does not have this issue because the `uart_rx_data` / `uart_rx_valid`
registers are written by the UART's own sequential logic (which does
not change `uart_rx_data` mid-load).

### 8.2 Software Breakpoints Only in RAM

Breakpoints work by overwriting instructions with `ebreak` /
`c.ebreak`. This only works for code resident in RAM. ROM-resident
code (the stub itself, or any flash-mapped code) cannot be
breakpointed.

**Workaround:** Place all code to be debugged in RAM. The debug
stub lives in ROM precisely so it never needs breakpoints on itself.

### 8.3 No Data Watchpoints

Hardware watchpoints require trigger comparators (CSRs + address
match logic) that are not implemented. Only code breakpoints are
supported.

### 8.4 No Hardware Single-Step

Single-step is implemented in software by next-PC prediction. The
stub decodes the current instruction, computes all possible
successor PCs, plants temporary `ebreak` instructions at each one,
and resumes. The CPU immediately hits one of them and re-enters the
stub.

**Implications:**

- Works correctly for all RV32IC instructions, including branches,
  `JAL`, `JALR`, `c.j`, `c.jr`, `c.beqz`/`c.bnez`.
- For branches, **both** the taken and fall-through targets get
  scratch breakpoints (the stub does not evaluate the condition).
- For `JALR`, the target depends on `rs1` — read from `g_regs[]`,
  which reflects the saved state at trap entry.
- Stepping over ROM-resident code is not possible (can't plant
  breakpoints there).
- Overhead: ~10 µs per step at 72 MHz.

### 8.5 `dbg_halt_req` Re-Trigger Prevention

The `dbg_halt_req` signal from the UART break detector is level-
sensitive (stays high while the RX line is low). Without protection,
it would re-trigger the trap immediately after `mret`, trapping the
CPU back into the stub in an infinite loop.

**Mechanism:** A 1-FF `dbg_halt_mask` register in the core:

- **Set** when `dbg_halt_req` is accepted (trap fires).
- **Cleared** on `mret` (return from trap handler).
- While set, further `dbg_halt_req` assertions are ignored.

The break detector's counter also self-clears when the RX line
goes high, but `dbg_halt_mask` is the primary protection.

### 8.6 `mepc` Overwrite on Debug Halt

`dbg_halt_req` saves `PC_new` (the next instruction) to `mepc`,
overwriting any previous `mepc` value. If the CPU is already inside
an ISR (e.g., handling an external IRQ), a debug halt will overwrite
the ISR's return address.

**Mitigation:** The ISR entry code in `crt0_stub.S` saves `mepc` to
`g_regs[]` immediately on entry, before any other `mepc` writes
can occur. The restore sequence writes `g_regs[pc]` back to `mepc`
before `mret`. This means the saved `mepc` in the register frame
always reflects the correct value at the time of the halt.

### 8.7 Address Space Doubling

The debug SoC uses `ADDR_WIDTH = RAM_AW + 1` to create a second
half for the ROM. This doubles the total address space compared to
a non-debug configuration. User firmware must be linked for the
wider address space (the lower half is still RAM, so existing code
typically just works).

### 8.8 `__sp_save` Placement Constraint

The ISR entry saves `sp` using zero-base addressing (`sw sp,
__sp_save(zero)`) before switching `sp` to the register frame. The
`__sp_save` variable must reside at a RAM address < 0x800 (the
signed 12-bit immediate range for `zero + offset`). The linker
script routes it to `.sp_scratch` at address 0x4.

### 8.9 Stub RAM Budget

The stub uses ~800 bytes of RAM for mutable data:

| Item | Size | Notes |
|---|---|---|
| `g_regs[]` (RV32I) | 132 bytes | 33 × 4 bytes (x0–x31 + PC) |
| `g_regs[]` (RV32E) | 68 bytes | 17 × 4 bytes (x0–x15 + PC) |
| Breakpoint table | 120 bytes | 10 slots × 12 bytes |
| `pkt_buf[]` | 280 bytes | RSP packet buffer |
| Stub stack | ~200 bytes | Grows down from top of RAM |
| `stepping` flag | 4 bytes | Single-step state |
| `__sp_save` | 4 bytes | Saved sp during ISR entry |

This is allocated in the upper portion of RAM (STUB_RAM section),
below the I/O registers. User firmware must not use this region.
The linker script's `ASSERT` checks for overflow into I/O space.

### 8.10 Simulation Speed

The Verilator bridge runs one RTL clock cycle per C++ loop iteration.
At typical Verilator speeds (~10–100 MHz simulation rate), each GDB
command takes milliseconds to execute — fast enough for interactive
use but noticeably slower than a real target.

The `+timeout=` plusarg prevents runaway simulations. The default
(100M cycles) is sufficient for typical interactive sessions. For
long-running `continue` operations, increase it:

```bash
build/gdb_bridge/obj_dir/Vattorv32_dbg +hex=fw.hex +timeout=1000000000
```

### 8.11 GDB `continue &` and Async Interrupt Timing

When GDB sends `continue`, the stub returns from the RSP loop and
the CPU resumes user code. To re-halt, GDB sends 0x03 (Ctrl-C) on
the TCP socket. The bridge converts this to a UART break (20 cycles
of RX low), which triggers `dbg_halt_req`. This is functionally
correct but may take a few hundred simulation cycles to take effect
(break counter saturation + trap latency).

In batch-mode GDB scripts, a `shell sleep 1` between `continue &`
and `interrupt` ensures the simulation has enough time to process
the halt.
